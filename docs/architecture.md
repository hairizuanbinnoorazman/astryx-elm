# Architecture

## Dependency policy

The published package may use official and community Elm packages. Dependencies
must be evaluated for API stability, maintenance, accessibility implications,
and package size. Likely foundation dependencies include:

- `elm/core` is mandatory.
- `elm/html` renders components and inline/theme styles.
- `elm/json` decodes keyboard and input events where `elm/html` helpers are not
  sufficient.
- `elm/browser` is allowed only for components that need global subscriptions or
  navigation integration.
- `elm/time` is allowed only for timed behavior such as toast expiry.
- `elm/svg` may be used for Elm-native icons and visual primitives.
- Community packages may be added when they remove substantial, well-tested
  work. Each addition must be documented and must not wrap a JavaScript runtime.

No npm package, imported JavaScript bundle, JavaScript-backed web component, or
hidden port module is part of core. The initial `elm.json` should include only
packages used by implemented modules rather than adding possible dependencies
preemptively.

## Public API shape

Follow the useful conventions in both Elm reference projects:

1. One public module per component or cohesive component family.
2. Opaque configuration and state types where invalid combinations matter.
3. Pipeline-friendly option functions for optional behavior.
4. Plain records for content that applications commonly construct or update.
5. `Html msg` slots for composition instead of a closed child hierarchy.
6. Complete module docs with `@docs` sections and a minimal working example.
7. Internal modules remain absent from `exposed-modules`.

A stateless component should generally look like:

```elm
Button.view
    [ Button.primary
    , Button.onClick Save
    , Button.disabled model.saving
    ]
    [ text "Save" ]
```

A stateful component should make ownership explicit:

```elm
type alias Model =
    { dialog : Dialog.State }

type Msg
    = DialogChanged Dialog.State

update msg model =
    case msg of
        DialogChanged state ->
            ( { model | dialog = state }, Cmd.none )

view model =
    Dialog.view
        { state = model.dialog
        , onStateChange = DialogChanged
        , title = text "Delete item?"
        , body = [ text "This cannot be undone." ]
        , footer = []
        }
```

Each interactive module decides whether it is:

- **Controlled:** state is entirely supplied by the application.
- **State machine:** the module exports opaque `State`, `Msg`, `init`, `update`,
  `view`, and optionally `subscriptions`; the parent maps child messages.

Do not expose two competing models for the same component in the first release.

## Styling and themes

Astryx's StyleX implementation and CSS custom-property themes become Elm data.
A `Theme` contains semantic tokens rather than raw component CSS. Components
render token-derived inline styles and semantic classes/attributes generated by
the package.

The first implementation should prefer CSS custom properties emitted on a
theme root. This preserves browser pseudo-classes, media queries, inheritance,
and efficient theme switching—capabilities that pure inline `style` attributes
cannot represent. The library itself emits the required CSS through an Elm view
or documented static asset generated in-repository; consumers do not install an
external stylesheet.

Token groups:

- semantic colors and status colors
- spacing and element sizes
- border widths and radii
- shadows and layer order
- typography roles and weights
- motion duration/easing with reduced-motion fallbacks

Component options should be semantic (`primary`, `danger`, `small`) rather than
accepting untyped token names everywhere. Escape hatches use explicit HTML
attributes, with documented precedence.

## Accessibility contract

- Prefer native controls (`button`, `input`, `select`, `dialog` where viable)
  before recreating interaction with ARIA.
- Every input supports a programmatically associated label, description, and
  error message.
- Keyboard behavior follows the relevant WAI-ARIA interaction pattern.
- Focus indication is always visible for keyboard interaction.
- Disabled and read-only semantics are distinct.
- Motion honors `prefers-reduced-motion`.
- Status is communicated with text or accessible names, never color alone.
- Component documentation identifies required labels and application-owned
  responsibilities.

## Testing and documentation

Borrow the split demonstrated by the references:

- Pure functions and state transitions receive `elm-test` unit tests.
- Rendered HTML is tested for element type, attributes, ARIA relationships, and
  emitted messages.
- A showcase application provides visual and keyboard test fixtures.
- `elm make --docs=/tmp/astryx-elm-docs.json` is a release gate.
- Browser tests are reserved for focus, keyboard, responsive, and overlay
  behavior that static HTML queries cannot prove.

Visual similarity to Astryx is a goal, but semantic HTML, keyboard behavior,
and stable Elm APIs are release blockers; pixel parity is not.
